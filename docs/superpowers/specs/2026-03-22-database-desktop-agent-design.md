# 数据库桌面管理器设计文档

## 1. 背景

本项目要做一个面向工程用户的桌面应用，用于管理和查询多种数据库，包括 `MySQL`、`PostgreSQL`、`MSSQL` 和 `ClickHouse`。

产品形态要求：

- 安装为桌面应用，可运行在 `macOS` 和 `Windows`
- 支持本地保存连接信息，并对凭据做加密存储
- 支持图形化浏览数据库对象、执行 SQL、查看结果、导出数据
- 支持通过 `chat` 方式交互，让 agent 帮助生成查询语句
- 默认允许只读查询自动执行
- 写操作、DDL、危险操作必须显式确认
- 先做单机本地使用，不做团队协作、共享连接或云同步

本设计文档采用 `Electron` 作为桌面壳方案。

## 2. 产品目标

### 2.1 核心目标

- 让后端工程师和数据工程师快速连接数据库
- 让用户更高效地浏览 schema、表、字段和查询结果
- 让用户通过自然语言快速生成可执行 SQL
- 在保证安全边界的前提下减少手写 SQL 的成本

### 2.2 成功标准

- 用户可以在 3 分钟内完成首次连接并看到库表
- 用户可以在一个界面里完成连接、查询、看结果和历史回看
- 用户可以用自然语言得到 SQL，并对只读查询实现自动执行
- 用户不会在无确认的情况下执行破坏性操作
- 应用在 macOS 和 Windows 上具备一致的主流程体验

## 3. 非目标

本 MVP 不包含以下能力：

- 团队协作
- 权限审批流
- 云端同步
- 数据资产治理
- BI 图表大屏
- 多人共享审计平台
- 复杂的数据建模或 ETL 编排

## 4. 目标用户

首要用户是后端工程师和数据工程师，典型场景包括：

- 查看生产或测试库结构
- 临时写 SQL 排障
- 验证数据是否符合预期
- 快速回答“某个业务字段当前是什么值”“这批数据分布如何”之类的问题

## 5. 方案选择

### 5.1 为什么选 Electron

Electron 的优势在于：

- 跨平台 UI 方案成熟
- 前端生态和数据库工具链更丰富
- 可直接使用大量 JavaScript 组件、编辑器和 agent 相关 SDK
- 打包、签名、自动更新、调试流程成熟

对于这个项目，最重要的是快速交付、生态可用和实现稳定，而不是最小安装包体积或最强原生性能。因此 Electron 更适合作为 MVP 底座。

### 5.2 为什么不是 Tauri

Tauri 的优势是体积小、资源占用低、Rust 核心更轻，但它对 Rust 工程能力、插件开发和系统 WebView 环境更敏感。对这个产品来说，Tauri 并非不能做，而是会增加前期实现和集成成本。

### 5.3 结论

本项目采用：

- `Electron` 负责桌面壳、窗口生命周期和前端 UI
- 本地业务核心负责连接管理、查询执行、agent 编排和安全控制
- 数据库适配层按数据库类型拆分

## 6. 总体架构

系统分为四层：

### 6.1 UI 层

负责展示和交互，不直接持有数据库协议细节。

包含：

- 连接列表
- 对象树
- SQL 编辑器
- 结果表格
- chat 面板
- 设置页
- 确认弹窗

### 6.2 应用服务层

负责编排业务流程。

包含：

- 连接生命周期管理
- 查询任务调度
- 查询历史管理
- agent 会话管理
- 权限与确认策略
- 错误归类与提示

### 6.3 数据访问层

负责与数据库建立连接并执行查询。

包含：

- MySQL 驱动适配
- PostgreSQL 驱动适配
- MSSQL 驱动适配
- ClickHouse 驱动适配
- 元数据读取
- SQL 执行
- 结果集标准化

### 6.4 安全与存储层

负责本地加密、敏感数据处理和审计记录。

包含：

- 凭据加密保存
- 应用锁定
- 操作日志
- 风险操作确认

## 7. Electron 架构设计

### 7.1 进程划分

采用标准 Electron 结构：

- `main process`
  - 创建窗口
  - 管理应用生命周期
  - 持有本地服务实例
  - 处理系统级能力
- `renderer process`
  - 承载 React 或同类前端 UI
  - 展示连接、编辑器、结果和 chat
- `preload`
  - 作为安全桥梁，向渲染层暴露受控 API
  - 禁止渲染层直接访问 Node 能力

### 7.2 通信方式

renderer 不直接操作数据库，也不直接访问文件系统。

所有敏感操作通过受控接口传递给 main process 或本地核心服务：

- 连接创建
- 查询执行
- 元数据读取
- 凭据保存
- 确认提交

### 7.3 原则

- 渲染层保持“薄”
- 数据库协议和敏感状态保留在本地核心层
- 所有危险动作都要经过统一策略判断

## 8. 核心模块

### 8.1 连接管理器

职责：

- 创建、编辑、删除连接
- 测试连接
- 持久化连接配置
- 管理连接分组和收藏

输入：

- 数据库类型
- host、port、database/schema、username、password
- 连接参数

输出：

- 可复用连接对象
- 连接状态
- 错误信息

### 8.2 对象浏览器

职责：

- 展示库、schema、表、视图、字段、索引等对象
- 提供展开、搜索和刷新
- 支持快速预览表数据

### 8.3 SQL 编辑器

职责：

- 提供多标签 SQL 编辑
- 支持语法高亮、补全和格式化
- 支持执行选中片段或当前脚本
- 展示执行耗时、错误信息和结果集
- 多 Tab/多连接状态管理：每个 SQL Editor/Chat Tab 都强绑定唯一的连接 Session，避免串库或上下文混乱

### 8.4 查询执行器

职责：

- 接收 SQL 并判断类型
- 根据连接执行查询
- 将不同数据库的结果标准化
- 记录执行时间和状态
- 支持长耗时查询的中断（Kill Query），允许用户手动取消慢查询

约束：

- 只读查询可直接执行
- 写操作必须经过用户确认
- 大结果集与防御性查询：对于未显式包含 `LIMIT` 的查询，自动注入 `LIMIT 1000`（或在游标层截断），并提示用户，防止大结果集压垮内存

### 8.5 查询历史

职责：

- 保存 SQL、连接、时间、执行状态、结果摘要
- 支持回放与复用
- 支持收藏常用查询

### 8.6 Chat Agent

职责：

- 理解用户自然语言问题
- 结合当前连接上下文生成 SQL
- 判断查询是否只读
- 解释 SQL 和执行结果

行为规则：

- 默认允许只读查询自动执行
- 任何可能修改数据或结构的操作必须确认
- 如果意图不明确，优先生成 SQL 并提示用户确认

### 8.7 安全控制器

职责：

- 判断 SQL 风险级别
- 管理确认弹窗
- 处理锁屏或重新解锁
- 记录审计日志

建议规则：

- `SELECT`、`SHOW`、`DESCRIBE`、`EXPLAIN` 归类为只读
- `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`、`CREATE`、`RENAME` 归类为危险操作
- 多语句执行默认按最高风险处理

### 8.8 本地安全存储

职责：

- 加密保存连接密码和敏感配置
- 保存最近使用记录、UI 状态和历史元数据

原则：

- 凭据必须加密后落盘，采用操作系统原生凭据管理器（如 macOS Keychain / Windows Credential Vault，可通过 Electron `safeStorage` 或 `keytar` 实现）
- 解密能力只在本机当前用户上下文内使用
- 不将明文密码暴露给渲染层

## 9. 数据库适配设计

每种数据库使用独立适配器，实现统一接口。

统一能力至少包括：

- 建立连接
- 断开连接
- 执行 SQL
- 获取 schema 元数据
- 获取表结构
- 获取样例数据

适配器差异处理：

- `MySQL` / `PostgreSQL` / `MSSQL` 的 schema、分页、参数语法不同
- `ClickHouse` 的查询和元信息接口与传统 OLTP 数据库不同
- 各数据库返回字段类型可能不一致，需要标准化映射

统一输出模型建议包括：

- `columns`
- `rows`
- `rowCount`
- `durationMs`
- `warnings`
- `error`

## 10. Chat Agent 设计

### 10.1 基础设施与隐私边界

- **模型接入**：底层依赖的大模型需用户自行配置 API 接入，系统同时兼容 OpenAI API 和 Anthropic API 标准。
- **隐私提醒**：在配置和使用时，需对用户进行明确的隐私边界提醒，告知哪些结构或数据将被发送至云端。

### 10.2 Schema 上下文与多表关联

为解决真实业务库表数量庞大及多表关联查询（如：“帮我查询 A 用户下了哪些订单，订单状态是怎么样的，用户余额还剩下多少钱”）的问题：

- **Schema 索引机制**：在本地对表结构建立轻量级索引。
- **按需注入**：Agent 在生成 SQL 前，先根据用户自然语言提取关键词，检索出最相关的若干张表（候选表），然后仅将这些相关表的 DDL 拼接进 Prompt，从而在不超 Token 限制的情况下精准完成多表 JOIN 查询的生成。

### 10.3 职责边界

agent 不是全权执行者，而是“受控查询助手”。

agent 可以：

- 读取当前连接上下文
- 生成 SQL
- 解释 SQL
- 分析只读结果
- 给出下一步建议

agent 不可以：

- 绕过确认直接执行破坏性 SQL
- 访问未授权连接
- 修改本地安全策略
- 自动扩展到团队共享场景

### 10.4 交互流程

1. 用户在 chat 输入自然语言问题
2. agent 识别意图和目标数据库上下文
3. agent 生成 SQL 和解释
4. 安全控制器判断 SQL 风险
5. 如果只读，自动执行并返回结果
6. 如果危险，弹出确认
7. 用户确认后执行并返回结果

### 10.5 Agent 输出格式

建议 agent 返回以下内容：

- 问题理解
- 生成的 SQL
- 风险说明
- 执行结果摘要
- 后续建议

这样可以避免 chat 变成黑盒。

## 11. 关键用户流程

### 11.1 首次使用

1. 用户启动应用
2. 创建数据库连接
3. 测试连接
4. 保存连接
5. 进入对象浏览器或 SQL 编辑器

### 11.2 直接写 SQL

1. 用户打开某个连接
2. 输入 SQL
3. 系统判断 SQL 类型
4. 只读则直接执行
5. 危险操作则要求确认
6. 展示结果并记录历史

### 11.3 使用 chat 查询

1. 用户输入问题
2. agent 生成 SQL
3. 系统判断风险
4. 自动执行只读查询
5. 危险查询要求确认
6. 返回结果和解释

## 12. 错误处理

### 12.1 连接失败

需要区分：

- 地址不通
- 认证失败
- 数据库不存在
- 权限不足
- TLS 或证书错误

前端必须给出可读提示，并提供重试入口。

### 12.2 SQL 执行失败

应返回：

- 错误信息
- 错误位置或数据库提示
- 执行耗时
- 是否已部分执行

### 12.3 Agent 失败

常见失败包括：

- 无法理解用户意图
- 无法找到合适的表或字段
- 生成 SQL 语法不正确
- 查询结果为空或过大

处理方式：

- 提供更明确的追问建议
- 允许用户直接编辑生成的 SQL
- 允许切换到手写 SQL 模式

## 13. 日志与审计

应用需要记录最小必要审计信息：

- 连接打开或关闭
- SQL 执行记录
- agent 生成 SQL 的内容
- 危险操作确认记录
- 错误和异常摘要

审计日志的目标是便于排障和回溯，不是做团队级合规平台。

## 14. 性能与体验要求

- 应用启动后应快速进入主界面
- 常用连接列表和历史要能快速加载
- 大结果集要避免一次性渲染过重
- 对对象树和历史列表做增量加载
- 多标签 SQL 编辑器要保持基本流畅

## 15. 安全要求

- 凭据必须本地加密保存
- 渲染层不能直接接触明文密码
- 危险 SQL 必须人工确认
- 支持应用锁定，避免别人直接打开看到连接信息
- 默认不接入云端，不上传数据内容

## 16. MVP 范围确认

第一阶段只做：

- 本地连接管理
- 多数据库驱动适配
- 对象浏览
- SQL 编辑器
- 查询结果展示
- 查询历史
- Chat Agent 生成和执行只读 SQL
- 危险操作确认
- 本地加密存储

暂不做：

- 团队协作
- 云同步
- 权限审批流
- 数据同步任务
- 图表分析
- 审计后台

## 17. 风险与取舍

### 17.1 风险：agent 误生成危险 SQL

缓解：

- SQL 风险分类器
- 统一确认弹窗
- 默认只读自动执行

### 17.2 风险：多数据库差异导致适配复杂

缓解：

- 统一抽象接口
- 每种数据库独立适配器
- 标准化结果模型

### 17.3 风险：Electron 包体较大

缓解：

- 这是可接受的 MVP 代价
- 先优化功能可交付性
- 后续若体积和占用成为硬约束，再评估迁移或局部替换

## 18. 后续演进方向

如果 MVP 验证成功，下一步可以考虑：

- 查询模板
- 智能补全增强
- 结果可视化
- 团队共享只读连接
- 本地脚本自动化
- 更多数据库适配
